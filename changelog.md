### Added (2026-07-16 20:50:33 UTC)
- **gmproxy diagnostics are now durable and surfaced in Export-GodModeLogs (menu option [11]); graceful-fallback PIDs are handed to the monitor for in-place SYSTEM elevation (no kill+relaunch duplicate).** Two improvements to the IFEO + gmproxy "born as SYSTEM" layer:
  - **`driver/gmproxy.c` now mirrors every `[GM-PROXY]` diagnostic line to a durable log file `%TEMP%\gmproxy.log` (append, UTF-8 via `GetTempPathW`), in addition to stderr.** IFEO launches `gmproxy.exe` detached (no console), so stderr was invisible on the VM -- the "acquired active-session SYSTEM token / relocated / no-usable-token WARN / graceful-fallback launch (PID) / SYSTEM success (PID+session) / launch-failure GLE" lines were never recoverable. New `DiagLog()` (variadic, writes both stderr and the log via `vfwprintf`) + `GmProxyDiagLogOpen()` (lazy, best-effort; logging never affects the launch). Each session is prefixed with a timestamped `=== gmproxy diag session <local time> ===` header. `stderr` writes are preserved so console launches still see output.
  - **`Export-GodModeLogs` (option [11] / `-DumpLogs`) now includes a new `===== GM-PROXY DIAGNOSTIC LOG =====` section** that reads `%TEMP%\gmproxy.log` (same `$env:TEMP` as the existing main/debug logs, which matches gmproxy's `GetTempPathW` since both run as the admin user) and dumps its full contents into the Desktop log bundle. A missing file is reported as `[No gmproxy log found] (expected at <path>)` -- mirroring the existing log-file guards -- so the dump never errors when no IFEO launch has occurred yet.
  - **Graceful-fallback PID handoff to the monitor for in-place elevation (no kill+relaunch duplicate).** When gmproxy's graceful fallback launches the target as the current user (because no session-correct SYSTEM token was stealable and `SeTcb` is unavailable), it now signals the God Mode monitor with that PID over a named pipe `\\.\pipe\GodMode-GmProxyFeedback` (new `SignalGmProxyFeedback()` -- best-effort, non-blocking `CreateFileW` + `OPEN_EXISTING`; if no monitor is listening the periodic scan remains the fallback, so this never regresses). The monitor side: new `Start-GmProxyFeedbackListener` runs a `NamedPipeServerStream` on a background `Start-ThreadJob` (with an explicit `PipeSecurity` ACL granting SYSTEM/Administrators full control and Everyone read/write so the admin-user-launched gmproxy can connect to the SYSTEM-created pipe; falls back to the default ACL if access-control types are unavailable), parses `PID=<decimal>` lines, and enqueues them into a synchronized `$script:GmProxyFeedbackQueue`. The `Start-Monitoring` loop drains that queue and calls new `Invoke-GmProxyFeedbackElevation`, which performs ONLY Phase 0 in-place token replacement (`[TokenOps]::ReplaceProcessTokenForPid`) -- no kill, no relaunch -- with a conservative shell/critical-name guard (defense-in-depth, since gmproxy's fallback only runs for IFEO-hooked normal programs). If in-place replacement fails, NO kill+relaunch is attempted here; the existing 15s periodic scan remains the safety net. This converges the fallback-launched app to SYSTEM directly instead of running as a user duplicate for up to 15s and then being killed+relaunched. `Stop-GmProxyFeedbackListener` is wired into `Disable-GodMode` for teardown parity with `Unregister-ProcessCreationWatcher`.
  - Regression coverage extended: `tests/Test-GmProxySession.ps1` adds source-level assertions for the durable log (`DiagLog`/`GmProxyDiagLogOpen`/`GetTempPathW`+`gmproxy.log`/append/`vfwprintf` x2), the named-pipe handoff (`SignalGmProxyFeedback`/`GodMode-GmProxyFeedback`/`PID=` payload/`CreateFileW`+`OPEN_EXISTING`/graceful-fallback calls it), the monitor listener + in-place elevation (`Start-GmProxyFeedbackListener`/`Stop-GmProxyFeedbackListener`/`Invoke-GmProxyFeedbackElevation`/`GmProxyFeedbackQueue`/`NamedPipeServerStream`/`ReplaceProcessTokenForPid`/`Start-Monitoring` starts+drains/`Disable-GodMode` stops), and Export-GodModeLogs surfacing the log (`GM-PROXY DIAGNOSTIC LOG`/`gmproxy.log`/missing-log guard). `tests/test-gmproxy-session.sh` adds source-invariant greps for the new C symbols (the MinGW cross-compile still proves the new code BUILDS and yields a PE binary). `tests/run-regressions.sh` comments updated additively. ### Fixed (2026-07-16 19:44:00 UTC)
- **Fixed IFEO/gmproxy-launched apps coming up ownerless / blank User column / unusable / instant-killed (Firefox, Chrome, other programs).** With the IFEO + gmproxy "born as SYSTEM" layer active, hooked apps launched in a broken ownerless state: Firefox showed a blank User column in Task Manager (could not tell SYSTEM vs admin), Chrome launched then was killed instantly, and other programs came up "unusable". Two interacting root causes, both fixed:
  - **`driver/gmproxy.c` stole a Session 0 SYSTEM token and could not relocate it to the interactive session.** IFEO launches `gmproxy.exe` AS THE INVOKING (admin) USER, which does NOT hold `SeTcbPrivilege`, so the existing `SetTokenInformation(TokenSessionId, 1)` call silently failed. When the named Session-1 SYSTEM sources (`winlogon.exe`/`dwm.exe`/`fontdrvhost.exe`) were Protected Process Light (PPL) and `OpenProcess` was denied, the old second pass fell back to ANY accessible SYSTEM process -- almost always a Session 0 one (`services.exe`/`svchost.exe`). The child was then born in Session 0 with no interactive desktop -> ownerless / unusable, exactly the reported symptom. Fix: `FindSystemProcessForToken` now takes the active console session and REQUIRES it. New helpers: `GetActiveConsoleSessionId` (dynamically loads `WTSGetActiveConsoleSessionId` from `wtsapi32.dll` -- no extra link dependency, identical MSVC/MinGW build), `IsProcessSessionId` (`ProcessIdToSessionId`), `IsOpenableSystemProcess` (openable + S-1-5-18; returns FALSE for PPL-blocked processes), and `StealSystemToken` (open + `DuplicateTokenEx` to a primary token). The finder prefers named interactive SYSTEM processes and falls back to the first openable SYSTEM process IN the active session (covers the all-named-sources-PPL case). `FindAnySystemProcessPid` + a best-effort `SeTcbPrivilege` enable add a relocate path (steal any SYSTEM token and `SetTokenInformation` it into the active session) that works when gmproxy itself runs as SYSTEM. **Graceful fallback**: if no session-correct SYSTEM token is openable and `SeTcb` is unavailable (the admin-launched case), gmproxy now launches the target AS THE CURRENT USER via the IFEO-bypass hardlink (`CreateProcessW`) instead of a broken ownerless Session-0 launch -- strictly better (the app works; the monitor still elevates it to SYSTEM via its normal kill+relaunch path). `WinSta0\Default` desktop, the `CreateHardLinkW` IFEO recursion bypass, the over-long path guard, and the `CreateProcessWithTokenW`/`LOGON_WITH_PROFILE` token path are all preserved.
  - **`God-Mode-Windows.ps1` monitor killed freshly-born IFEO apps during the brief WMI owner-resolution window.** `Stop-NonSystemInstances` and both `Invoke-ParallelElevation` kill sites (sequential + threadjob) killed any instance whose `Invoke-CimMethod GetOwner` returned a non-SYSTEM owner. In the sub-second window right after an IFEO/gmproxy launch, WMI often returns a BLANK/unresolvable owner for the new logon session -- which read as "non-SYSTEM" and got the instance killed (Chrome instant-kill). Fix: all three kill sites now require a KNOWN non-SYSTEM owner (`if ($owner -and $owner.User -and $owner.User -ne "SYSTEM")`) -- blank/unresolvable owners are left alone. (`Get-NonSystemProcessesParallel` already guarded with `$owner -and` and is unchanged.) In non-IFEO mode this adds at most a ~500ms-1s delay before a genuinely admin-owned app is killed+relaunched; in IFEO mode it stops the monitor from fighting the born-as-SYSTEM layer.
  - Regression coverage: new `tests/Test-GmProxySession.ps1` (22 source-level assertions: gmproxy.c session-correctness invariants + graceful fallback + regression guards, and the monitor blank-owner guard in `Stop-NonSystemInstances` + both `Invoke-ParallelElevation` sites) and new `tests/test-gmproxy-session.sh` (MinGW cross-compile of `gmproxy.c` proving the new code BUILDS and yields a PE binary + source invariants; FAIL SUMMARY (N) + exit 1). Both wired into `tests/run-regressions.sh` (steps 3 and 5); `tests/Test-IfeoElevation.ps1` section 8 extended additively with session-awareness + graceful-fallback assertions. Full `tests/run-regressions.sh` passes 11/11, FAIL SUMMARY 0; `syntax_check.ps1` passes 12/12 (0 warnings, 0 failed); `gmproxy.c` MinGW-compile verified clean.

### Fixed (2026-07-16 18:33:00 UTC)
- **Fixed Task Manager (and shell-launched apps) not elevating to SYSTEM.** Symptom: Task Manager launched from the shell showed blank usernames in the User column (not `SYSTEM`), i.e. the child was never born as SYSTEM. Two independent root causes, both fixed:
  - **`driver/gmhook.c` `TryCreateProcessWithSystemToken` bailed out unelevated whenever the caller passed a `STARTUPINFOEX` / `EXTENDED_STARTUPINFO_PRESENT` (`0x00080000`) structure.** `explorer.exe` launches Task Manager (and most modern shell apps) with `STARTUPINFOEX`, so the `CreateProcessW` IAT hook silently fell through to the real unelevated `CreateProcessW` and the child never became SYSTEM. The old guard existed to avoid the original `0xC0000005` (caused by leaving `cb == sizeof(STARTUPINFOEX)` on a truncated stack `STARTUPINFOW` so the kernel read a garbage `lpAttributeList`). The guard is now a **DOWNGRADE** instead of a skip: copy only the first `sizeof(STARTUPINFOW)` base bytes (a `STARTUPINFOEX` begins with an embedded `STARTUPINFOW`, so those bytes are always valid), clamp `cb` to `sizeof(STARTUPINFOW)`, and CLEAR the `EXTENDED_STARTUPINFO_PRESENT` bit in the creation flags passed on. The child is now BORN as SYSTEM via `CreateProcessWithTokenW`; the caller's attribute list is intentionally dropped (inherited-handle list / mitigations), and on any failure the existing fall-back to the real `CreateProcessW` still runs (unelevated but no crash) — no regression. The `0xC0000005` root cause (kernel reading past the plain `STARTUPINFOW`) is still prevented by the `cb` clamp + EXTENDED-bit clear.
  - **`God-Mode-Windows.ps1` `-LaunchTaskMgrAsSystem` always failed and fell back to an unelevated `Start-Process`.** It called `[TokenOps]::CreateProcessAsSystem` -> `CreateProcessAsUser`, which requires `SeAssignPrimaryTokenPrivilege` — a privilege an interactive Administrator token does NOT hold (UAC-filtered), so it failed with Win32 error 1314 (`ERROR_PRIVILEGE_NOT_HELD`) every time and dropped to `Start-Process $TaskMgrCopy` (unelevated). Switched to `[TokenOps]::CreateProcessFromToken` -> `CreateProcessWithTokenW`, which only requires `SeImpersonatePrivilege` (held by an interactive Administrator token). Also resolved the real `%WINDIR%\System32\taskmgr.exe` as the launch target: the legacy `$GodModeInstallDir\taskmgr_real.exe` copy is only ever DELETED (in `Block-TaskManager` / `Unblock-TaskManager`) and is never created anywhere in the script, so `Test-Path` on it was almost always false and the SYSTEM launch was silently skipped. Best-effort ensures the `seclogon` (Secondary Logon) service is running first, since `CreateProcessWithTokenW` is serviced by `seclogon` and a stopped/disabled `seclogon` (common on tweaked/VM builds) makes every stolen-token launch fail with Win32 1460/1058. `CreateProcessAsSystem`/`CreateProcessAsUser` remains correct when the caller is already SYSTEM (used by the scheduled-task monitor via `Invoke-ParallelElevation`); it is only wrong for this admin-CLI launcher.
  - Auto-elevation flag forwarding: `-LaunchTaskMgrAsSystem` is now forwarded to the elevated child in the auto-elevation block (it was missing from the `$ArgsString` list), so `God-Mode-Windows.ps1 -LaunchTaskMgrAsSystem` actually reaches the handler after re-launching as Administrator instead of dropping the flag.
  - Regression coverage updated: `tests/Test-GmHookFix.c` `shouldTakeTokenPath` now mirrors the DOWNGRADE decision (extended `STARTUPINFOEX` and `EXTENDED_STARTUPINFO_PRESENT` now TAKE the token path with `cb` clamped to `sizeof(STARTUPINFOW)` and the EXTENDED bit cleared, instead of falling back), and adds `outFlagsOut` assertions that the EXTENDED bit is always cleared. `tests/Test-GodModeCrashFix.ps1` source assertions updated to require the base-`sizeof` copy + `~0x00080000` EXTENDED-bit clear (downgrade) instead of the old `cb > sizeof` skip. Full regression suite (`tests/run-regressions.sh`) passes: 7/7, FAIL SUMMARY 0.

### Fixed (2026-07-15 18:59:00 UTC)
- **Fixed PowerShell 7 (`pwsh.exe`) fatal crash `0xC0000005` (STATUS_ACCESS_VIOLATION) inside `Kernel32.CreateProcess`.** Root cause: `gmhook.dll` was being injected into shell/launcher hosts (PowerShell, cmd, terminals) and rerouting their `CreateProcessW` calls through the stolen-token `CreateProcessWithTokenW` path. PowerShell/cmd launch native commands via `CreateProcessW` as their core job; in-process IAT hooking of those calls destabilizes the host with a native access violation that PowerShell `try/catch` cannot recover from (it kills `pwsh.exe` — exactly the reported crash). A CLR-level AV in `CreateProcess` is not catchable by PowerShell (noted at `God-Mode-Windows.ps1` Disable-SecurityAuditing).
  - `driver/gmhook.c`: Added `IsShellLauncherProcess()` matching `pwsh.exe`, `powershell.exe`, `cmd.exe`, `wt.exe`, `conhost.exe`, `OpenConsole.exe`, `WindowsTerminal.exe`. ORed with `IsCriticalProcess` at the three guard sites: `HookCreateProcessW` base-name pass-through, `GetMsgProc` auto-install, and `DllMain` auto-install. These hosts are still elevated to SYSTEM via `Invoke-HybridElevation` / `CreateProcessAsSystem`; they are simply not IAT-hooked in-process.
  - `God-Mode-Windows.ps1` `Install-ProcessHook`: Added the same host names (bare, no `.exe`, since `Get-Process.Name` returns them that way) to the `$CriticalProcs` DLL-injection skip-list so `gmhook.dll` is never loaded into PowerShell/cmd/terminals. `Uninstall-ProcessHook` unchanged.
  - `driver/gmhook.c` `HookCreateProcessW`: Fixed recursion-guard leak — the re-entry path returned without resetting `inHook` to 0, so a single re-entry permanently left `inHook==1` and silently bypassed elevation on every subsequent `CreateProcessW` call. Now resets `inHook` on both return paths.
  - `driver/gmhook.c`: Documented why MinGW builds rely on deterministic STARTUPINFO validation (EXTENDED_STARTUPINFO_PRESENT bypass, `cb` clamp, NULL PI/token guards) instead of SEH for the token path.
